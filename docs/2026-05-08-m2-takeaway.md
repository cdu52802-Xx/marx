## 关键决策与 plan 修订

### 决策 1 · 走方案 b 离线缓存（关键路径变更）⭐

**触发**：takeaway 坑 12 — 用户本机 https://query.wikidata.org/ 打不开（中国大陆典型网络环境）。本机不能挂代理，但**第三机能挂代理 + 能登 Wikidata**（不是 Codex 主力机，是第三台浏览/办公机）。

**决策路径**：
1. 4 选 1 fallback（a 挂代理 / b 离线缓存 / c 切 Codex / d 代跑）
2. 选 b 因为本机无法挂代理 + c 风险大（Codex 机大概率同 ISP 也不通）+ d 不可持续
3. 三阶段分工：阶段 1（Claude 主力机写 Task 1-5）→ 阶段 2（第三机网页 UI 跑 5 个 SPARQL）→ 阶段 3（Claude 主力机跑 Task 6-9）

**长期红利**：cache JSON 入 git 后整个项目自包含，任何 fork 出去的人 clone + `MARX_USE_CACHE=1 npm run fetch:skeleton` 就能复现 39 节点骨架，不再依赖某个国外 endpoint 可达性。M3 阶段 C 拉 SEP 等其他国外 endpoint 时同套路可复用。

### 决策 2 · 第三机零安装策略（PM-friendly 设计）

**问题**：原方案让 user 在第三机 git clone 仓库 + 装 git。

**修订**：scripts/sparql 通过 GitHub 公仓暴露 → 第三机只用浏览器（GitHub raw + query.wikidata.org）+ 微信文件传输助手回传 5 个 JSON。零工具链安装。

**适用场景**：未来 M3 阶段 C 拉 SEP 等数据源时，user 仍可走"第三机浏览器 + 网盘回传"路径，不需在第三机配开发环境。

### 决策 3 · plan 修订点：cache 开关加 ~30 行

Plan v1.1 Task 5 fetch-skeleton.ts 是基于"挂代理"假设写的，没有 cache 开关。M2 实施时加了：

- `USE_CACHE = process.env.MARX_USE_CACHE === '1'` 环境变量开关
- `runQueryFromCache(sparqlFile)` 读 `scripts/sparql/cache/<num>.json` 替代 endpoint fetch
- `flatToBinding(flat)` 把网页 UI 下载的 simplified flat array 转成内部 SparqlBinding 格式
- 双格式兼容（simplified flat array + 标准 SPARQL Result）

未来挂代理时 unset `MARX_USE_CACHE` 直接走 endpoint，**同一份脚本 a/b 两种方案都能用**。

### 决策 4 · m1-skeleton.ts 删除推迟到 Task 8

**原因**：Plan Task 7 Step 4 让删 m1-skeleton.ts，但 relations.test.ts 在 Task 8 才改。如果 Task 7 commit 时删了 m1-skeleton，Task 7 的 commit 状态会让 test broken（违反 atomic commit "每个 commit 至少能 build + test pass"）。

**修订**：m1-skeleton.ts 删除推迟到 Task 8 跟 relations.test.ts 改写一起。每个 commit 状态干净。

### 决策 5 · "基督教的本質" 误进 person 类（known issue 不修）

## 哪些坑值得记下来？

### 坑 13 · Wikidata 网页 UI "JSON" vs "JSON（详细）" 格式差异

**症状**：在 query.wikidata.org 网页 UI 跑查询点 Download，"JSON" 选项实际下载的是 simplified flat array（`[{"key":"value",...},...]`），**不是**标准 SPARQL Result format（`{head, results: {bindings: [...]}}`）。后者要选 "JSON（详细）" 才能拿到。

**最初的错误推断**：Claude 主力机写 fetch-skeleton.ts 时假设 cache 文件是标准 SPARQL Result 格式（直接读 `json.results.bindings`），跟 user 沟通时也建议选普通 "JSON" 而非 "JSON（详细）"——这两个判断是矛盾的：选普通 JSON 但脚本期望详细 JSON。

**修复路径**：
1. user 拿回 5 个 JSON 后我 sanity check 发现是 simplified flat array
2. 加 `flatToBinding` 转换函数 + `runQueryFromCache` 双格式兼容
3. 让 user 重下 5 次 vs 加 ~15 行脚本——明显后者 ROI 高

**M3+ 防御**：写涉及"网页 UI 下载格式"的脚本时，**先在网页 UI 实际下载 1 个文件确认格式**，再写解析逻辑。不要靠 W3C 标准猜默认下载是哪个格式。

### 坑 14 · prettier 全局 format 顺手改无关文件

**症状**：跑 `npm run format` 修 fetch-skeleton.ts 格式问题时，prettier 把 5 个 mockup HTML（`public/styles/*.html`）也格式化了——纯排版改动（CSS 单行 → 多行 selector block）零语义变化，但跟 M2 无关。

**修复路径**：
- `git restore public/styles/` 撤销 mockup 改动（destructive 命令需 user 显式授权）
- M2 commit 链保持干净

**M3+ 防御**：跑 `npm run format` 之前先 `git status` 看作用域，或者用 `npx prettier --write src/ scripts/` 限定范围，避免 prettier 全局扫描顺手改 mockup / docs / specs。

### 坑 15 · `git restore` 是 destructive 命令，agent 默认无权限

**症状**：尝试 `git restore public/styles/` 撤销 mockup 改动被权限拒，报错 "irreversibly discards uncommitted local changes... without explicit user authorization"。

**修复**：跟 user 显式同步选项（A 撤销 / B 单独 commit / C 一起 push），user 回 "A" 后再跑。

**M3+ 防御**：destructive git 操作（`reset --hard` / `restore` / `checkout --` / `clean -f` / `branch -D`）写 plan 时**预先标记需要 user 授权**，不要假设 agent 能直接跑。

### 坑 16 · Windows + Bash 中文 commit message 仍走 -F 文件方式

**确认**：memory `workflow_cross_machine.md` 的 "中文 commit message 走 -F 文件方式" 在 M2 9 个 commit 全部生效。Git Bash for Windows 用 `-m "中文"` 在某些终端配置下仍会乱码，`-F .commit-msg.tmp` 100% 可靠。

**操作模式（已成稳定）**：
1. `Write F:\AI\projects\Marx\.commit-msg.tmp`（含 commit message）
2. `git add ... && git commit -F .commit-msg.tmp && rm .commit-msg.tmp`

`.commit-msg.tmp` 已被 `.gitignore` 默认（不显式列但 untracked 不会进 commit）。

### 坑 17 · 阶段间断切换协作的 visual companion 缺失

**症状**：阶段 2 user 切到第三机时，Claude 在 Claude 主力机 idle 等 user 回来。中间没有"我现在在做什么 / 下一步是什么"的 visual companion。user 多次需要主动问"我现在应该做什么 / 是要切第三机吗"。

**深层原因**：分阶段协作里有"上下文切换断点"——user 切机器时 Claude 不知道 user 已经做到哪一步、需要什么具体指引。

**M3+ 防御**：
- 每个阶段开始前给 user **白话版 step-by-step**（含具体 URL / 按钮位置 / 文件命名）
- 阶段切换时主动给"对照表"：你做什么 / 我做什么 / 现在卡在哪
- 等 user 回来时给一个明确的"回来后告诉我什么"prompt 模板（"5 个 JSON 拿回来了，存在 [路径]"）
